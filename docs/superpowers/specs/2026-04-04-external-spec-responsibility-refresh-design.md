# external_spec.md 責務再整理 設計
作成日: 2026-04-04

## 背景

現状の [docs/external_spec.md](../../../docs/external_spec.md) は、
外部設計書としての役割に対して、次の情報が混在している。

- 利用者・本アプリ・外部要素の責務
- 機能要件や状態遷移
- 実装寄りの構成要素や将来像

一方、現行実装では次の前提が確認できる。

- デスクトップ配布形態は WebView ベースである
- セットアップ完了前後で利用者向け画面が切り替わる
- 録画、編集、アップロードは本アプリが外部要素を連携して進行する

このため、外部設計書の先頭付近で責務境界を明確にしないと、
後続の構成図や機能要件の主語がぶれやすい状態になっている。

## 目的

- [docs/external_spec.md](../../../docs/external_spec.md) を現行実装準拠で最新化する
- `利用者`、`本アプリ`、`外部要素` の責務境界を明確にする
- 以降の章が責務境界に従って読めるよう、章立てを再整理する

## 非目的

- 内部設計書レベルで本アプリ内部を詳細分解すること
- 未実装機能を外部仕様として先行定義すること
- 利用ガイドや開発ガイドの責務までこの文書へ取り込むこと

## 前提

- 最新化基準は「現行実装準拠」とする
- 外部設計書上の主体は `利用者`、`本アプリ`、`外部要素` の 3 区分とする
- 外部設計書上では `本アプリ` を 1 つの責務主体として扱う
- `GUI`、`Web API`、`WebView` は主体として独立させず、
  `本アプリが利用者へ提供する接点` または `実装方式` として扱う

## 採用方針

### 方針 A: 主体別責務を先に定義する

最初に `誰が何を担うか` を固定し、その後に構成図と機能要件を置く。
これにより、各節の主語がぶれにくくなる。

### 方針 B: システム境界の内側は本アプリのみとする

`OBS Studio`、`FFmpeg`、`Tesseract`、`Groq`、`YouTube`、各ハードウェアは、
すべてシステム境界の外側に配置する。
本アプリは、それらを利用して利用者向け機能を提供する主体として表現する。

### 方針 C: 本アプリ内部の実装分解は外部設計書へ持ち込まない

現行実装には WebView、FastAPI、Svelte などの構成要素があるが、
外部設計書では本アプリ内部の構造としては扱わない。
外部設計書では、利用者から見た機能提供主体としてまとめて記述する。

### 方針 D: 担わない責務も明示する

責務の境界を明確にするため、
`本アプリが担わないこと` も明文化する。
これにより、外部要素依存の範囲が読み取りやすくなる。

## 主体別の責務定義

### 利用者の責務

- Switch、キャプチャデバイス、マイク、YouTube 連携に必要な機材・アカウント・認証情報を用意する
- セットアップウィザードに従って初期設定を完了する
- 必要に応じて録画後データの確認、メタデータ修正、アップロード実行を行う

### 本アプリの責務

- セットアップ状態に応じて、利用者へ適切な画面と操作導線を提供する
- 録画準備、録画制御、メタデータ管理、編集、アップロードを統合的に進行する
- 外部要素の処理結果を受け取り、利用者へ状態、進捗、エラー、成果物を提示する
- 設定、録画ファイル、字幕、編集済みファイルなどの管理対象を保持する

### 外部要素の責務

- Switch はゲーム映像を出力する
- キャプチャデバイスとマイクは、本アプリが利用可能な入力を提供する
- OBS Studio は映像取り込みと録画実行基盤を提供する
- FFmpeg、Tesseract、Groq、YouTube は、それぞれ専用機能を提供する
- Windows と WebView 実行環境は、デスクトップ実行基盤を提供する

### 本アプリが担わない責務

- ゲーム映像そのものの生成
- OBS や YouTube など外部製品の内部仕様保証
- 外部サービス障害や認証ポリシー変更の恒久解消

## 責務分担表

| 主体 | 担う責務 | 担わない責務 |
| --- | --- | --- |
| 利用者 | 機材・アカウント・認証情報の準備、初期設定、必要時の確認や修正 | 録画制御や外部サービス連携の内部処理 |
| 本アプリ | セットアップ案内、録画準備、録画制御、メタデータ管理、編集、アップロード、状態表示、エラー通知 | ゲーム映像生成、外部製品の内部仕様保証、外部障害の恒久解消 |
| Switch | ゲーム映像の出力 | 録画や保存の制御 |
| キャプチャデバイス | 映像を PC へ入力可能な形で中継 | 録画状態やメタデータの管理 |
| マイク | 音声入力の提供 | 文字起こしや字幕化 |
| OBS Studio | 映像入力の取り込み、録画実行基盤の提供 | 本アプリの業務判断、動画整理、アップロード |
| FFmpeg | 動画・音声の編集処理 | 編集対象の業務判断 |
| Tesseract / Groq | OCR、文字起こしなどの専用処理 | 録画制御、保存、公開判断 |
| YouTube | 動画公開、字幕付与、プレイリスト管理の受け皿 | ローカル動画編集、録画データ管理 |
| Windows / WebView 実行環境 | デスクトップ実行基盤の提供 | 業務機能の実行判断 |

## システム境界の表現ルール

- システム境界の内側は `本アプリ` のみとする
- `OBS Studio`、`FFmpeg`、`Tesseract`、`Groq`、`YouTube`、各ハードウェアは境界の外側に置く
- 本アプリが外部要素へ何を依頼し、何を受け取るかを入出力として明示する
- `GUI` や `Web API` は独立主体として描かず、`本アプリ` の利用者接点として扱う

## external_spec.md の想定章立て

1. 目的
   現行実装準拠で、本アプリが現在提供している価値を簡潔に示す。
2. スコープ
   対象ユーザ、対応環境、扱う外部要素を整理する。
3. システム境界と責務分担
   利用者、本アプリ、外部要素、本アプリが担わない責務を明記する。
4. 外部連携構成
   ハードウェア構成、ソフトウェア構成、主要な入出力の流れを責務境界に沿って示す。
5. 利用者から見た主要機能
   録画準備、録画、編集、アップロード、設定、確認・修正を利用者視点で整理する。
6. 制約・前提条件
   必要な外部ソフト、認証、未対応範囲、利用条件を分離して記載する。

## 表現ルール

- 文書の主語は原則として `利用者`、`本アプリ`、`外部要素` のいずれかに固定する
- 実装で未確認の事項は断定しない
- 未対応機能は `未対応` または `現状未対応` と明記する
- 外部設計書では内部実装方式の詳細を主題にしない

## 成功条件

- 外部設計書を読んだ人が、利用者、本アプリ、外部要素の責任範囲を区別できる
- 構成図と機能要件の主語が責務境界と整合する
- `docs/usage.md` と `docs/DEVELOPMENT.md` の責務を侵食しない
- 現行実装と矛盾する記述を減らせる

## 検証方法

- 各章の主語が責務境界に沿っているかを確認する
- 構成図でシステム境界内外の配置が一貫しているかを確認する
- `docs/usage.md` と `docs/DEVELOPMENT.md` と比較し、文書責務の重複がないかを確認する

## リスク

- 既存の `docs/external_spec.md` には別作業中の変更が含まれている可能性がある
  - 対策: 本体反映時は既存差分を確認し、責務整理と競合しない範囲で統合する
- 現行実装と旧仕様の差分が残っており、責務以外の節でも再整理が必要になる可能性がある
  - 対策: まず責務境界を固定し、その後に構成図と機能要件を順に見直す

## 結論

[docs/external_spec.md](../../../docs/external_spec.md) は、
現行実装準拠の外部設計書として、
`利用者`、`本アプリ`、`外部要素` の責務境界を先に定義する構成へ改める。

そのうえで、構成図、主要機能、制約・前提条件を
責務境界に従って再配置することで、文書全体の一貫性を高める。
