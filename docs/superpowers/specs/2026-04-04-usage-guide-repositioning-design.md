# usage.md 位置づけ再整理 設計
作成日: 2026-04-04

## 背景

現状の [docs/usage.md](../../../docs/usage.md) は、利用者向けガイドでありながら、
以下の 2 系統の情報を同時に抱えている。

- 配布版ユーザー向けの利用手順
- ソースコードからのビルドや手動セットアップを含む開発者寄り手順

一方、現行の [README.md](../../../README.md) はすでに次の導線へ更新されている。

- GitHub Releases から `SplatReplay.zip` を取得する
- `SplatReplay.exe` を起動する
- 初回はセットアップウィザードに従って準備する

また、実装上も PyInstaller 配布とセットアップウィザードが前提になっており、
旧 `usage.md` に残っている `git clone`、`task.exe build`、Python / uv / Node.js / Task の導入説明は、
現状の利用フローと一致しない。

## 目的

- [docs/usage.md](../../../docs/usage.md) を「配布版ユーザー向けの詳細ガイド」として再定義する
- 配布 ZIP 展開後からセットアップ完了、日常利用開始までを迷わず辿れる文書にする
- 開発者向け情報との責務重複を減らし、文書全体の役割分担を明確にする

## 非目的

- 開発環境構築手順を `usage.md` に残すこと
- 外部仕様書レベルの詳細な処理フローを `usage.md` に再掲すること
- セットアップウィザード未実装の新手順を文書で先行定義すること

## 制約

- 対象読者は配布版利用者とし、開発者は主対象にしない
- コード上で確認できる現行のセットアップステップに沿って記述する
- 未確認の挙動は断定せず、推測で補完しない
- 既存の [docs/DEVELOPMENT.md](../../../docs/DEVELOPMENT.md) と
  [docs/external_spec.md](../../../docs/external_spec.md) の責務を侵食しない

## 文書責務の再定義

### `docs/usage.md` で扱う内容

- この文書の対象読者と責務
- 配布版の入手と起動
- 利用前に必要な機材・アカウント・外部ソフトの概要
- セットアップウィザード各ステップで求められる準備と判断
- 普段の利用方法
- 利用者向けカスタマイズ
- 利用者向けトラブルシューティング

### `docs/usage.md` で扱わない内容

- `git clone`、`task.exe build`、`task.exe run` などの開発・ビルド手順
- Python / uv / Node.js / Task などの開発用依存の導入
- ビルド成果物ディレクトリや中間生成物の説明
- 外部仕様書レベルの詳細な状態遷移やメタデータ抽出仕様

## 他文書との責務分担

- プロジェクト概要と最短導線:
  [README.md](../../../README.md)
- 開発環境構築、ビルド、検証、日常開発フロー:
  [docs/DEVELOPMENT.md](../../../docs/DEVELOPMENT.md)
- 外部仕様、処理フロー、機能要件:
  [docs/external_spec.md](../../../docs/external_spec.md)

`usage.md` 冒頭で、配布版利用者向け文書であることと、
開発参加時は `docs/DEVELOPMENT.md` を参照することを明示する。

## 採用方針

### 方針 A: 配布版ユーザー中心に再構成する

章立ては「入手する」「準備する」「セットアップする」「使う」「困ったときに確認する」の順で構成する。
これにより、利用者が実際に辿る導線に沿った文書になる。

### 方針 B: セットアップウィザード前提の説明へ置き換える

外部ソフトの導入を 1 つずつ手作業で説明するのではなく、
セットアップウィザードで求められる内容を中心に整理する。
詳細なインストール手順は必要最小限に留め、必要なら公式サイト参照へ誘導する。

### 方針 C: 詳細仕様は別文書へ逃がす

録画・編集・アップロードの内部仕様は簡潔な概要だけを残し、
詳細は `docs/external_spec.md` に委譲する。
これにより `usage.md` が「使い方の文書」である状態を維持できる。

## 想定構成

1. この文書の対象
   配布版利用者向けであり、開発者向け手順は対象外であることを明記する。
2. 利用開始までの流れ
   `SplatReplay.zip` の展開、`SplatReplay.exe` の起動、初回セットアップ開始までを案内する。
3. 事前に必要なもの
   Windows 11、Switch、キャプチャーボード、OBS Studio、必要に応じたマイク、必須の YouTube アカウントなどを整理する。
4. セットアップウィザード
   現行実装のステップに沿って次を案内する。
   - ハードウェア確認
   - FFmpeg
   - OBS Studio
   - Tesseract OCR
   - フォント
   - 文字起こし
   - YouTube
5. 使い方
   通常起動、録画から編集・アップロードまでの大まかな流れ、停止方法を記載する。
6. カスタマイズとトラブルシューティング
   サムネイル差し替えなどの利用者向け設定と、最低限の障害対応をまとめる。

## 現行情報の棚卸し方針

### 残す情報

- 配布版の入手方法と起動方法
- 利用者に必要な外部ソフト・機材の概要
- セットアップウィザードで求められる準備内容
- 基本的な利用フロー
- 利用者向けカスタマイズ
- 利用者向けトラブルシューティング

### 削る情報

- `git clone`
- `task.exe build`
- Python / uv / Node.js / Task のインストール
- `dist/SplatReplay/` や `backend/dist/` を前提にした説明
- 詳細仕様寄りの章立て

### 別文書へ逃がす情報

- 開発・ビルド手順は `docs/DEVELOPMENT.md`
- 詳しい外部仕様や処理フローは `docs/external_spec.md`

## 想定する表現ルール

- 手順は「ユーザーが何をするか」を主語にして書く
- 詳細な手動インストール手順は原則として簡略化する
- セットアップウィザードで吸収される判断は、ウィザード前提で説明する
- 未確認の事項は書かない。必要なら「詳細は画面表示に従う」と表現する

## 成功条件

- `usage.md` を読めば、配布版ユーザーがソースコードやビルド手順を知らなくても導入開始できる
- `usage.md` から開発者向け手順が除かれ、`docs/DEVELOPMENT.md` との責務境界が明確になる
- セットアップウィザード追加後の利用導線と文書内容が整合する
- 文書全体の流れが、現在の README の最短導線と矛盾しない

## 検証方法

- `usage.md` の記載対象が配布版利用者向けに限定されているかを確認する
- `README.md`、`docs/DEVELOPMENT.md`、`docs/external_spec.md` と責務重複がないかを確認する
- セットアップウィザードの現行ステップと記載内容の対応を確認する

## リスク

- セットアップウィザード実装と文書の粒度がずれる可能性がある
  - 対策: 実装上のステップ名に沿って説明し、未確認の詳細は書かない
- 詳細説明を削りすぎると、初回利用者が外部ソフト準備で迷う可能性がある
  - 対策: 必要な外部ソフトと準備物は一覧化し、詳細は公式サイトやウィザードに委ねる

## 結論

[docs/usage.md](../../../docs/usage.md) は、
「配布版ユーザーが ZIP 展開後にセットアップウィザードを通じて利用開始するための詳細ガイド」
として再構成する。

これに伴い、旧来の手動ビルド・開発依存インストール・詳細仕様説明は削除または他文書へ委譲し、
配布版利用フローに沿った実用的な文書へ整理する。
