# GitHub Release 作成スキル設計

作成日: 2026-03-28

## 背景

このリポジトリでは GitHub Release を `gh` で作成できるが、毎回同じ確認と手順を人手で組み立てる必要がある。特に、前回リリースからの差分把握、ユーザー向けリリースノート草案化、`draft` / `prerelease` 判定、ビルド成果物の zip 添付は手順漏れや判断ぶれが起きやすい。

既存の公開運用では release / tag は `v0.4.0` 形式で付いている。一方、ユーザー入力は `1.2.3` 形式を想定しているため、入力形式と実タグ形式の差も吸収する必要がある。

## 目的

- `1.2.3` 形式の入力から、プロジェクト運用に沿った GitHub Release 作成を安全に実行できるようにする。
- バージョン未指定でも、前回リリースと差分素材から次バージョン候補を提案できるようにする。
- 前回リリースからの差分をもとに、コミット一覧ではなくユーザー向け要約としてリリースノート本文を作成できるようにする。
- 実行前に必ずユーザー承認を挟み、誤タグ付けや誤公開を防ぐ。
- `task.exe build` の成果物を zip 化し、draft release に添付できるようにする。

## 成功条件

- スキルは `1.2.3` を受け付け、実際のタグ名とリリース名を `v1.2.3` に補正できる。
- `prepare` をバージョン未指定で実行した場合、patch / minor / major の候補を返せる。
- 前回リリースを GitHub Releases から優先取得し、取得失敗時のみタグを代替利用できる。
- リリースノート本文は、エージェントが「仕様追加・変更」「不具合修正」「ユーザー対応が必要な内容」の区分で箇条書き作成できる。
- 実行前に、予定タグ名、予定リリース名、`draft` / `prerelease` 判定、ノート本文、ビルドと zip 添付予定をユーザーへ提示し、許可がない限り書き込み操作を行わない。
- 許可後に `HEAD` へタグ付けし、`task.exe build` 実行後に `dist/SplatReplay` を zip 化して draft release に添付できる。

## 非目標

- 自動公開まで行うこと。release は常に draft で作成する。
- コミット一覧や差分パッチをリリースノート本文に含めること。
- 既存の Taskfile や CI に release 専用タスクを追加すること。
- semver 以外の複雑なバージョン体系へ広く対応すること。

## 検討した案

### 案 A: スキル本体は対話と判定を持ち、実処理は同梱スクリプトへ寄せる

採用案。スキルはユーザーとのやり取り、前提確認、承認ゲート、異常系判断を担当し、実際の release 作成手順は再利用可能なスクリプトに集約する。

- 利点: 再現性が高い。タグ判定、ノート生成、build / zip / upload の処理を固定化しやすい。
- 欠点: スクリプト実装とテストが必要になる。

### 案 B: スキルは手順書だけにして、その場でコマンドを組み立てる

不採用。実装量は減るが、自然言語解釈の揺れが大きく、今回のような承認ゲートや差分要約まで含む運用では事故率が高い。

### 案 C: Taskfile に release タスクを追加し、スキルは呼び出し役だけにする

不採用。運用入口としては便利だが、プロジェクト本体のビルド定義まで変更対象が広がる。今回の主目的は Codex スキル化であり、影響範囲が過大である。

## 採用方針

`.codex/skills/create-github-release/` を新設し、以下を含める。

- `SKILL.md`: トリガー条件、対話手順、許可ゲート、失敗時の停止条件を記載する。
- `scripts/create_github_release.py`: リリース境界取得、差分収集、タグ存在確認、build / zip / draft release 作成を実行する。
- 必要なら `references/` にプロジェクト固有ルールや出力形式を分離する。

スキルはプロジェクトスコープとし、このリポジトリ内の `.codex/skills` に配置する。

## 入力と正規化

### 入力形式

- ユーザー入力は `1.2.3` を正とする。
- `v1.2.3` が渡された場合は、想定入力との差分として検知する。
- バージョン未指定も許容する。この場合は実行ではなく提案フローへ入る。

### 正規化方針

- 内部処理用の正規化済みバージョンは `1.2.3` とする。
- 実際に作成するタグ名とリリース名は `v1.2.3` とする。
- 既存タグ運用と異なる形式が検出された場合は、未確認のまま進めず、補正案を提示してユーザー確認を取る。
- バージョン未指定時は、前回リリースを基準に patch / minor / major 候補を返し、エージェントが推奨版を選ぶ。

## 前回リリース判定

優先順位は以下のとおりとする。

1. `gh release list` で取得できる既存 Release を優先する。
2. Release が取得できない場合のみ、既存タグ一覧から semver 風タグを代替利用する。

既存実データ上は `v0.4.0`, `v0.3.0` などの prerelease が存在する。新バージョンのメジャー番号が `0` の場合は、今回作成分も `prerelease` とする。

## バージョン提案方針

引数なし `prepare` では、前回リリースを基準に次の 3 候補を返す。

- patch: `X.Y.(Z+1)`
- minor: `X.(Y+1).0`
- major: `(X+1).0.0`

候補の算術自体はスクリプトで行う。どの候補を推奨するかはエージェントが差分を読んで判断する。

推奨ルールは以下のとおりとする。

- 不具合修正のみ、または使い勝手が変わらない内部変更だけなら patch
- ユーザーにとって仕様追加・変更があるなら minor
- 互換性破壊、移行、再設定、運用変更などユーザー対応が必要なら major

## リリースノート生成方針

前回リリースから `HEAD` までのコミットと変更ファイル傾向をもとに、エージェントが本文をユーザー向け箇条書きとして作成する。本文にコミット一覧は含めない。

基本構成は以下の 3 区分とする。

- 仕様追加・変更
- 不具合修正
- ユーザー対応が必要な内容

該当項目がない区分は省略する。分類はコミットメッセージと差分内容をもとにエージェントが行う。完全自動判定は誤り得るため、実行前に草案本文をそのままユーザーへ提示する。

### ユーザー対応が必要な内容の判定観点

- 設定ファイルやフォルダ構成の変更
- 操作手順の変更
- 互換性影響や移行作業の有無
- 再ビルド、再設定、再配置、再インストールが必要か

## 実行フロー

### 許可取得前

1. バージョン入力を検証し、必要なら補正候補を提示する。
2. 前回リリースを特定する。
3. 前回リリースから `HEAD` までの差分を取得する。
4. バージョン指定がない場合は、patch / minor / major 候補を計算する。
5. リリースノート作成に必要な差分素材を取得する。
6. エージェントがリリースノート草案と推奨バージョンを作成する。
7. 予定タグ名、予定リリース名、`draft` / `prerelease` 判定、候補バージョン、推奨理由、ノート本文、build / zip / 添付予定をユーザーへ提示する。
8. ユーザーの明示的許可を待つ。

この段階では読み取り系操作のみを行い、タグ作成・push・Release 作成は行わない。

### 許可取得後

1. 作業ツリーがクリーンか確認する。
2. 同名タグ、同名 Release の既存有無を確認する。
3. ユーザーが確定したバージョンで `HEAD` に `vX.Y.Z` タグを作成する。
4. `task.exe build` を実行する。
5. `dist/SplatReplay` を `SplatReplay-vX.Y.Z.zip` に圧縮する。
6. `gh release create` で draft release を作成し、zip を添付する。
7. メジャーバージョンが `0` の場合のみ `--prerelease` を付ける。

## 成果物

- タグ名: `vX.Y.Z`
- リリース名: `vX.Y.Z`
- 添付 zip 名: `SplatReplay-vX.Y.Z.zip`
- 圧縮対象: `dist/SplatReplay`

## 安全条件と停止条件

以下のいずれかに該当する場合は停止し、ユーザーへ状況を報告する。

- バージョン形式が想定外で、補正方針が確定できない。
- バージョン未指定なのに、候補と推奨理由を提示しないまま実行へ進もうとした。
- 作業ツリーに未コミット変更があり、ビルド成果物とタグ対象 `HEAD` の一致が保証できない。
- 同名タグまたは同名 Release が既に存在する。
- `task.exe build` が失敗する。
- zip 作成に失敗する。
- Release 作成または asset upload に失敗する。

## 実装上の前提

- Windows 環境のため、build は `task.exe build` を使う。
- 既存 Taskfile により、成果物は `dist/SplatReplay` に集約される。
- GitHub CLI は認証済みで、`repo` 権限を持つことを前提とする。

## テスト方針

最低限、以下を確認する。

- `1.2.3` 入力が `v1.2.3` に正規化されること。
- バージョン未指定時に前回リリースから patch / minor / major 候補を計算できること。
- メジャーバージョン `0` のときだけ `prerelease` 判定になること。
- 既存 Release がある場合に前回境界を正しく取得できること。
- ノート生成結果が 3 区分のうち必要な見出しだけを出すこと。
- 実行前フェーズでは書き込み系操作を行わないこと。
- タグ重複、Release 重複、dirty worktree で停止すること。
- build 成果物の zip 名と添付対象が期待どおりであること。

## リスクと緩和策

- 差分からの自動分類は誤判定があり得る。
  - 緩和策: 実行前に本文草案を必ず表示し、ユーザー許可を必須とする。
- 既存タグ運用との不一致で意図しない命名になる恐れがある。
  - 緩和策: 入力と実タグ名を分離し、補正結果を事前表示する。
- build に時間がかかる、または環境差異で失敗する可能性がある。
  - 緩和策: build は許可後にのみ実行し、失敗時は Release 作成前で止める。
- 作業ツリーが dirty のまま進めると、タグ対象と成果物がずれる。
  - 緩和策: 許可後処理の最初に dirty チェックを行い、問題があれば停止する。

## 実装順の提案

1. スキル雛形を `.codex/skills/create-github-release/` に作成する。
2. スクリプトでバージョン正規化、前回リリース取得、差分素材収集、候補バージョン算出を実装する。
3. スクリプトでタグ重複確認、dirty チェック、build / zip / draft release 作成を実装する。
4. `SKILL.md` に、本文作成はエージェントが担当すること、対話フロー、停止条件を記述する。
5. スキルの検証と前方テストを行う。

## 未確認事項

- エージェントによる差分分類と推奨バージョン判断は、コミットメッセージ品質と変更内容の読み取りに依存するため、どの程度まで安定化できるかは暫定である。
- zip 作成コマンドは Windows での可搬性を優先して PowerShell `Compress-Archive` を第一候補とする想定だが、実装時に検証が必要である。
